## 4. 结构视角

### 4.1 业务逻辑层的分解

业务逻辑层的开发包图参见软件体系结构文档。

#### 4.1.1 discussionbl

##### (1)模块概述

discussionbl模块承担的需求参见需求规格说明文档功能需求及相关非功能需求。

discussionbl模块的职责及接口参见软件体系结构说明文档。

##### (2)整体结构

根据体系结构的设计，我们将系统分为展示层、业务逻辑层、数据层。每一层之间为了增加灵活性，我们会添加接口。例如，为了隔离展示层和业务逻辑层，我们添加了discussionController，并且在discussionController和具体的业务逻辑实现之间都依赖接口，降低了耦合性。PostPO和CommentPO、ReplyPO是作为帖子、评论、回复数据的持久化对象被添加到设计模型中去的。

![](https://b2.kuibu.net/file/imgdisk/imgs/2021/07/d2cd05b1d00a3e4e.png)

discussionbl模块的各个类的职责如下表所示：

| 模块            | 职责                                       |
| --------------- | ------------------------------------------ |
| PostController  | 负责接收前端请求，调用相应的业务逻辑       |
| PostService     | 负责定义相应的业务逻辑                     |
| PostServiceImpl | 负责实现创建与查看帖子及其评论所需要的服务 |
